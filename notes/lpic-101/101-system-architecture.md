## Topic 101.1: Determine and Configure Hardware Settings

Strictly speaking, the operating system is just the kernel and its components which control the hardware and manages all processes. It is common, however, to use the term “operating system” more loosely to refer to the entire software stack that runs on top of the kernel, including the user-space applications and utilities.

Every hardware part requires a software component to control the corresponding device. This software component is called a **kernel module** and it can be part of the official Linux kernel or added separately from other sources. Linux kernel modules related to hardware devices are also called **drivers**, as in other operating systems.

The commands `lspci`, `lsusb` and `lsmod` act as front-ends to read hardware information stored by the operating system.

### Device Inspection

List all devices currently connected to the **Peripheral Component Interconnect (PCI)** bus:

```shell
lspci
```

or, for more detailed information of the device with the hexadecimal ID **02:00.0**:

```shell
lspci -s 02:00.0 -v
```

lsusb : lists Universal Serial Bus (USB) devices currently connected to the machine

```shell
lsusb
```

or, for more detailed information of the device with the ID **80ee:0021**:

```shell
lsusb -d 80ee:0021 -v
```

or to list the device mappings as a hierarchical tree as well as the corresponding **kernel module** (driver):

```shell
lsusb -t
```

The `lsmod` command is used to show all the currently loaded kernel modules. It is common to have a large set of loaded kernel modules in a standard Linux system at any time.

```shell
lsmod
```

The `modprobe` command is used to add or remove a kernel module from the Linux kernel. For example, to remove the kernel module named **snd-hda-intel**:

```shell
modprobe -r snd-hda-intel
```

To add the kernel module named **snd-hda-intel**:

```shell
modprobe snd-hda-intel
```

The `modinfo` command is used to display information about a specific kernel module. For example, to display information about the kernel module named **snd-hda-intel**:

```shell
modinfo snd-hda-intel
```

The `nouveau modeset=0` option can be added to the file 
`/etc/modprobe.d/nouveau.conf` to disable the `modeset` kernel feature.

The file `/etc/modprobe.d/blacklist.conf` (or even better manually create `/etc/modprobe.d/<module_name>.conf`) can be used to block the loading of the module if the module is causing problems.

The information read by `lspci`, `lsusb` and `lsmod` is stored in special files in the directories `/proc` and `/sys`. These directories are mount points to filesystems that are created by the Linux kernel at boot time. The files in these directories are not real files on disk, but rather virtual files in RAM space that provide a way for the kernel to expose information about the system to user-space applications.

`/proc` and `/sys` are referred to as **pseudo filesystems** because they do not contain real files on disk, but rather virtual files that are generated by the kernel at runtime. These files provide a way for user-space applications to access information about the system and to interact with the kernel.

### Information Files

Important files in the `/proc` directory include:

- `/proc/cpuinfo`: contains information about the CPU, such as the model, speed, and number of cores.
- `/proc/meminfo`: contains information about the system's memory, such as the total amount of RAM and the amount of free memory.
- `/proc/uptime`: contains the amount of time the system has been running since the last boot. This is displayed in seconds, with the first number representing the total uptime and the second number representing the amount of time the system has been idle.
- `/proc/version`: contains information about the Linux kernel version and the compiler used to build it.
- `/proc/modules`: contains a list of all currently loaded kernel modules, along with their size and the number of instances currently in use.
- `/proc/interrupts`: contains a list of all the interrupts currently being handled by the system, along with the number of times each interrupt has been triggered.
- `/proc/ioports`: contains a list of all the I/O ports currently in use by the system, along with the name of the device that is using each port.
- `/proc/partitions`: contains a list of all the partitions currently recognized by the system, along with their size and the name of the device that they are associated with.
- `/proc/dma`: contains a list of all the Direct Memory Access (DMA) channels currently in use by the system, along with the name of the device that is using each channel.

### Device Files

Device files are special files that represent hardware devices on a Linux system. They are typically located in the `/dev` directory and are used by the operating system to interact with hardware devices.
For example, the device file `/dev/sda` represents the first hard disk on the system, while `/dev/ttyS0` represents the first serial port.

Device files are created and managed by the Linux kernel, and they provide a way for user-space applications to access hardware devices without needing to know the details of how the hardware works.

**Coldplug detection** is a feature of the Linux kernel that allows it to detect and configure hardware devices that are present at boot time. When the system boots, the kernel scans the hardware and creates device files for each detected device in the `/dev` directory. This process is known as coldplug detection because it occurs during the boot process when the system is "cold" and has not yet fully initialized.

**Hotplug detection** is a feature of the Linux kernel that allows it to detect and configure hardware devices that are added or removed while the system is running. When a new device is added to the system, the kernel detects it and creates a corresponding device file in the `/dev` directory. When a device is removed, the kernel removes the corresponding device file. This process is known as hotplug detection because it occurs while the system is "hot" and fully initialized.

The operating system's main component, the **Linux kernel**, is loaded into memory by a program called a **bootloader**.

The bootloader is responsible for initializing the hardware and loading the kernel into memory. The bootloader also provides a way for the user to select which operating system to boot if multiple operating systems are installed on the same machine.

The most common bootloader used on Linux systems is called **GRUB** (GRand Unified Bootloader). GRUB is a powerful and flexible bootloader that can be configured to boot a wide variety of operating systems, including Linux, Windows, and macOS.

When the system is powered on, the BIOS or UEFI firmware initializes the hardware and then looks for a bootloader on the storage devices. Once the bootloader is found, it takes over the boot process and loads the kernel into memory. The kernel then initializes the rest of the system and starts the user-space applications.

The initialization of the operating system starts when the bootloader loads the kernel into RAM. Then, the kernel will take charge of the CPU and will start to detect and setup the fundamental aspects of the operating system, like basic hardware configuration and memory addressing.

The kernel will then open the `initramfs` (initial RAM filesystem). The initramfs is an archive containing a filesystem used as a temporary root filesystem during the boot process. The main purpose of an initramfs file is to provide the required modules so the kernel can access the “real” root filesystem of the operating system.

As soon as the root filesystem is available, the kernel will mount all filesystems configured in `/etc/fstab` and then will execute the first program, a utility named `init`. The init program is responsible for running all initialization scripts and system daemons. There are distinct implementations of such system initiators apart from the traditional init, like `systemd` and `Upstart`. Once the init program is loaded, the initramfs is removed from RAM.

In recent years most major Linux distributions have gradually adopted `systemd` as their default system manager.

### BIOS and UEFI

The **BIOS**, short for **Basic Input/Output System**, is a program stored in a **non-volatile memory chip attached to the motherboard**. It is executed every time the computer is powered on.
This type of program is called **firmware** and its storage location is separate from the other storage devices the system may have.
The BIOS assumes that the first **440 Bytes** in the first storage device are the first stage of the bootloader (also called **bootstrap**).

The first **512 Bytes** of a storage device are named the **MBR (Master Boot Record)** of storage devices using the standard DOS partition schema and, in addition to the first stage of the bootloader, contain the partition table.
If the MBR does not contain the correct data, the system will not be able to boot, unless an alternative method is employed.

The pre-operating steps to boot a system using the BIOS firmware are:

1. The POST (power-on self-test) process is executed to identify simple hardware failures as soon as the machine is powered on.
2. The BIOS activates the basic components to load the system, like video output, keyboard and storage media.
3. The BIOS loads the first stage of the bootloader from the MBR (the first 440 bytes of the first device, as defined in the BIOS configuration utility).
4. The first stage of the bootloader calls the second stage of the bootloader, responsible for presenting boot options and loading the kernel.

The **UEFI (Unified Extensible Firmware Interface)** is a modern replacement for the BIOS firmware. It provides a more flexible and powerful interface between the operating system and the hardware. UEFI supports larger storage devices, faster boot times, and more advanced security features than BIOS.

The UEFI does not rely on the MBR, taking into account only the settings stored in its non-volatile memory (NVRAM) attached to the motherboard. UEFI compatible programs are called Extensible Firmware Interface (EFI) applications and they are stored in a special partition called the **EFI System Partition (ESP)**. The ESP is a FAT32 formatted partition that contains the bootloader and other necessary files for booting the system.

The pre-operating steps to boot a system using the UEFI firmware are:

1. The POST (power-on self-test) process is executed to identify simple hardware failures as soon as the machine is powered on.
2. The UEFI activates the basic components to load the system, like video output, keyboard and storage media.
3. UEFI’s firmware reads the definitions stored in NVRAM to execute the pre-defined EFI application stored in the ESP partition’s filesystem. Usually, the pre-defined EFI application is a bootloader.
4. If the pre-defined EFI application is a bootloader, it will load the kernel to start the operating system.

The UEFI standard also supports a feature called Secure Boot, which only allows the execution of signed EFI applications, that is, EFI applications authorized by the hardware manufacturer. This feature increases the protection against malicious software, but can make it difficult to install operating systems not covered by the manufacturer’s warranty.

The most popular bootloader for Linux in the x86 architecture is **GRUB (Grand Unified Bootloader)**. As soon as it is called by the BIOS or by the UEFI, GRUB displays a list of operating systems available to boot.

GRUB can be configured to boot multiple operating systems, including different Linux distributions, Windows, and macOS. It also provides a command-line interface for advanced users to troubleshoot boot issues. GRUB can be installed on the MBR of a storage device or on the ESP partition in UEFI systems. The configuration file for GRUB is typically located at `/boot/grub/grub.cfg` and can be edited to customize the boot menu and other settings.

Settings related to the boot process, like the default operating system to boot and the timeout for the boot menu, can be configured in the GRUB configuration file. For example, to set the default operating system to boot, you can edit the `GRUB_DEFAULT` variable in the configuration file:

```shell
GRUB_DEFAULT=0
```

`acpi=off` is a kernel parameter that can be added to the GRUB configuration file to disable the Advanced Configuration and Power Interface (ACPI) support in the Linux kernel. ACPI is a power management feature that allows the operating system to control the power state of the hardware devices. Disabling ACPI can help to resolve certain hardware compatibility issues, but it may also cause some devices to stop working properly.

`init=/bin/bash` is a kernel parameter that can be added to the GRUB configuration file to specify an alternative init process to be executed instead of the default init system. This can be useful for troubleshooting and recovery purposes, as it allows you to boot into a minimal environment with a shell prompt. When this parameter is used, the system will boot directly into a bash shell, bypassing the normal initialization process. This can be helpful for diagnosing and fixing issues with the system, but it should be used with caution, as it can also lead to data loss if not used properly.

`systemd.unit=rescue.target` is a kernel parameter that can be added to the GRUB configuration file to specify that the system should boot into rescue mode. Rescue mode is a special mode of operation that allows you to troubleshoot and repair issues with the system. When this parameter is used, the system will boot into a minimal environment with a shell prompt, similar to the `init=/bin/bash` parameter, but with some additional services and features enabled to help with troubleshooting and recovery.

### Initialization Inspection

Errors that occur during the boot process can compromise the system's expected behavior. To investigate such errors, the `dmesg` command can be used to display the kernel ring buffer, which contains messages generated by the kernel during the boot process and while the system is running. The `dmesg` command can be used to identify hardware issues, driver problems, and other errors that may have occurred during the boot process.

```shell
dmesg
```

or, for a human-readable and paginated output with timestamps:

```shell
dmesg --human
```

In systems based on `systemd`, command `journalctl` will show the initialization messages with options `-b`, `--boot`, `-k` or `--dmesg`.

Command `journalctl --list-boots` shows a list of boot numbers relative to the current boot, their identification hash and the timestamps of the first and
last corresponding messages.

`journalctl -D /mnt/hd/var/log/journal` or `journalctl --directory=/mnt/hd/var/log/journal` can be used to inspect the journal files of an offline machine, assuming its mount point is `/mnt/hd`.

## Topic 101.3: Change runlevels / boot targets and shutdown or reboot system

Candidates should be able to manage the `SysVinit` runlevel or `systemd` boot target of the system. This objective includes changing to single user mode, shutdown or rebooting the system. Candidates should be able to alert users before switching runlevels / boot targets and properly terminate processes. This objective also includes setting the default `SysVinit` runlevel or `systemd` boot target. It also includes awareness of Upstart as an alternative to `SysVinit` or `systemd`.

Learning outcomes:

- Set the default runlevel or boot target.
- Change between runlevels / boot targets including single user mode.
- Shutdown and reboot from the command line.
- Alert users before switching runlevels / boot targets or other major system events.
- Properly terminate processes.
- Awareness of `acpid`.

### Introduction

A common feature among operating systems following Unix design principles is the employment of separate processes to control distinct functions of the system. These processes, called daemons (or, more generally, services), are also responsible for extended features underlying the operating system, like network application services (HTTP server, file sharing, email, etc.), databases, on-demand configuration, etc.

Daemons are typically started at boot time and run in the background, without any user interaction. Examples of daemons include web servers, database servers, and print servers.

The service manager is the component of the operating system responsible for starting, stopping, and managing the daemons. It provides a way for users and administrators to control which daemons are running on the system and to configure their behavior. The most common service manager used on Linux systems is `systemd`, which has become the default service manager for many Linux distributions. Other service managers include `SysVinit` and `Upstart`.

The service manager is the first user-space program to be executed by the kernel during the boot process. It is responsible for starting all the necessary daemons/services required for the system to function properly. The process identification number (PID) of the service manager is always 1, as it is the first process to be executed by the kernel.

### SysVinit

The traditional service manager for Linux systems is called `SysVinit`. It uses a system of **runlevels** to define the state of the system and which daemons should be running in each runlevel. Runlevels are numbered 0 to 6. The default runlevel is typically 3 (multi-user mode with networking), but it can be changed to other runlevels, such as 1 (single-user mode) or 5 (graphical mode).

- Runlevel 0: System shutdown.
- Runlevel 1, s or single: Single user mode, without network and other non-essential capabilities (maintenance mode).
- Runlevel 2, 3 or 4: Multi-user mode. Users can log in by console or network. Runlevels 2 and 4 are not often used.
- Runlevel 5: Multi-user mode. It is equivalent to 3, plus the graphical mode login.
- Runlevel 6: System restart.

### systemd

`systemd` is a modern service manager that has become the default for many Linux distributions used to manage system resources and services, which are called **units** in `systemd`. Units can be of different types, such as services, sockets, devices, mounts, etc. `systemd` uses a system of **targets** to define the state of the system and which units should be running in each target. Targets are similar to runlevels in `SysVinit`, but they are more flexible and can be customized to fit the needs of the system.

There are seven distinct types of `systemd` units:

1. `service`: A unit that manages a system service.
2. `socket`: A unit that manages a socket, which is a communication endpoint used by services to listen for incoming connections.
3. `device`: A unit that manages a device, such as a hardware component or a virtual device.
4. `mount`: A unit that manages a filesystem mount point.
5. `automount`: A unit that manages an automount point, which is a special type of mount point that is automatically mounted when accessed.
6. `target`: A unit that manages a target, which is a group of units that are started together to achieve a specific state of the system.
7. `snapshot`: A unit that manages a snapshot of the system state, which can be used to restore the system to a previous state.

The `systemctl` command is used to control the `systemd` service manager. It can be used to start, stop, and manage units, as well as to change the default target and to switch between targets. For a fictitious unit called `unit.service`, the following commands can be used:

- To start the unit:

```shell
systemctl start unit.service
```

- To stop the unit:

```shell
systemctl stop unit.service
```

- To restart the unit:

```shell
systemctl restart unit.service
```

- To check the status of the unit:

```shell
systemctl status unit.service
```

- To show *active* if the unit is active, or *inactive* if it is not:

```shell
systemctl is-active unit.service
```

- To enable the unit to start at boot time:

```shell
systemctl enable unit.service
```

- To disable the unit from starting at boot time:

```shell
systemctl disable unit.service
```

- To verify if the unit starts at boot time:

```shell
systemctl is-enabled unit.service
```

- To list all active units:

```shell
systemctl list-units --type=service --state=active
```

There are corresponding targets to `SysV` `runlevels`, starting with `runlevel0.target` up to `runlevel6.target`. To change the default target, the following command can be used:

```shell
systemctl set-default multi-user.target
```

To determine the current default target:

```shell
systemctl get-default
```

### Upstart

`Upstart` is an alternative service manager that was used in some Linux distributions before `systemd` became the default. It uses a system of **jobs** to define the state of the system and which daemons should be running in each job. Jobs are similar to targets in `systemd`, but they are less flexible and can be more difficult to manage.

The `initctl` command is used to control the `Upstart` service manager. It can be used to start, stop, and manage jobs, as well as to change the default job and to switch between jobs. Upstart was developed for the Ubuntu Linux distribution to help facilitate parallel startup of processes. Ubuntu has stopped using Upstart since 2015 when it switched from Upstart to `systemd`.

### Shutdown and Restart

Command shutdown acts as an intermediary to `SysV` or `systemd` procedures, that is, it executes the requested action by calling the corresponding action in the services manager adopted by the system.

- To shutdown the system immediately:

```shell
shutdown now
```

- To shutdown the system in 10 minutes:

```shell
shutdown +10
```

- To shutdown the system at a specific time, for example, at 11:00 PM:

```shell
shutdown 23:00
```

- To restart the system immediately:

```shell
shutdown -r now
```

- To restart the system in 10 minutes:

```shell
shutdown -r +10
```

- To notify users about the shutdown or restart, a message can be added to the command. For example, to shutdown the system in 10 minutes with a custom message:

```shell
shutdown +10 "The system will shutdown in 10 minutes for maintenance. Please save your work."
```

The `systemctl` command can also be used to shutdown or restart the system. For example, to shutdown the system immediately:

```shell
systemctl poweroff
```

To restart the system immediately:

```shell
systemctl reboot
```
